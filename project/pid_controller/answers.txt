//    The Python goal-ramp (0.05 m/frame) prevents instantaneous CTE jumps that would

//    spiral-waypoint jitter, keeping the output smooth throughout.
//
// 3. Throttle Velocity Error (bottom-left): Error starts at −3.0 m/s (vehicle at rest,
//    target 3 m/s) then converges to near-zero within 200 frames.  Steady-state error ≈

//    Small transient spikes (±0.1 m/s) appear at replanning cycles but are corrected
//    within a few frames.
//
// 4. Throttle/Brake Commands (bottom-right): Throttle holds at 0.383 average (below the
//    CARLA 1.0 cap). The initial burst reaches 0.908 (frames 0–50, vehicle from rest),
//    then settles into cruise. Brake is never applied (0 frames with brake > 0.01).
//    With Kd=0.0 for throttle there is no derivative noise amplification.

// --- What is the effect of the PID according to the plots, how each part of the PID affects the control command? ---
//
// Proportional (Kp): Provides the primary corrective action proportional to the current
//   error. For steering, Kp=0.05 generates a correction proportional to the cross-track
//   error, pulling the vehicle toward the planned spiral path. For throttle, Kp=0.3
//   drives acceleration proportional to the speed deficit (vel_err = actual − target).
//   The initial throttle burst (0.908) at frame 0 came directly from Kp × (−3.0 m/s) =
//   0.3 × 3 = 0.9, demonstrating pure proportional action from rest.
//
// Integral (Ki): Corrects persistent steady-state bias. Ki=0 for steering prevents
//   windup through NPC detour manoeuvres (a non-zero integral would build a lane-bias
//   residual that overshoots back after the NPC is cleared). Ki=0.05 for throttle proved
//   decisive: without it the vehicle cruised 0.7 m/s below target; with it, steady-state
//   velocity error converges to ≈ −0.003 m/s.
//
// Derivative (Kd): Anticipates the error trend and opposes it. For steering, Kd=0.25
//   damps lateral oscillation by sensing the rate of CTE change and reducing the
//   correction before overshoot. An EMA filter (α=0.25) is applied to the raw derivative
//   to suppress frame-to-frame spiral-waypoint jitter (Kd/dt ≈ 7.5 at 30 Hz would
//   otherwise amplify 0.01 m noise into 0.075 rad steer pulses every frame). Without
//   the goal-ramp and EMA, a single-frame 3.5 m CTE jump produced d_raw=102, steer
//   clamped to 1.2 for 32 consecutive frames. For throttle, Kd=0.0 deliberately avoids
//   amplifying velocity sensor noise into throttle chatter.

// --- How would you design a way to automatically tune the PID parameters? ---
//
// Twiddle (coordinate ascent):
//   1. Define a scalar score: total accumulated |CTE| over a fixed evaluation window.
//   2. For each gain [Kp, Ki, Kd]: try gain += dp, run N steps, record score.
//      If score improves keep the change and grow dp *= 1.1.
//      Otherwise try gain -= 2*dp; if that improves keep it, else revert and shrink dp *= 0.9.
//   3. Repeat until sum(dp) < tolerance.
//
// Alternative — Ziegler-Nichols step response:
//   Apply a step input, measure ultimate gain Ku and oscillation period Tu, then compute
//   Kp = 0.6*Ku, Ki = 2*Kp/Tu, Kd = Kp*Tu/8.
//
// In production, Bayesian optimisation (Gaussian Process surrogate) is preferred
//   because it finds good gains in fewer simulation episodes than Twiddle.

// --- PID controller is a model free controller, i.e. it does not use a model of the car.
//     Could you explain the pros and cons of this type of controller? ---
//
// Pros:
//   • No vehicle model required — works on any car with zero re-identification effort.
//   • Simple to implement and interpret; only 3 gains to tune per channel.
//   • Robust to model mismatch (tire wear, payload, road surface changes).
//   • Low computational cost — runs well within real-time budget.
//
// Cons:
//   • Purely reactive — acts only on current error, cannot anticipate upcoming curves
//     or predict the effect of a control action over a future horizon.
//   • Integral windup: if the vehicle is stuck or physically constrained, Ki accumulates
//     a large bias that causes aggressive action once the constraint is released.
//   • Gain sensitivity: a single gain set cannot be optimal across all speeds and
//     road conditions; separate tuning or gain scheduling is needed.
//   • No feedforward: a known reference trajectory cannot be used to pre-correct;
//     all corrections are error-driven and therefore always lag the reference.

// --- (Optional) What would you do to improve the PID controller? ---
//
// 1. Anti-windup clamping: freeze Ki accumulation whenever the output is saturated,
//    preventing the integral from growing beyond what actuators can deliver.
// 2. Goal-ramp smoothing (already implemented): lane-change target moves 0.05 m/frame
//    toward the new lane instead of jumping instantly, eliminating the derivative kick
//    that previously caused 32 sign-flip oscillations over 8 m.
// 3. Derivative EMA filter (already implemented, α=0.25): suppresses high-frequency
//    CTE jitter without delaying response to genuine errors; steer max dropped from
//    1.2 (clamped) to 0.187 during lane changes.
// 4. Gain scheduling: use a lookup table indexed by speed (high Kp at low speed for
//    parking maneuvers, lower Kp at highway speeds to avoid oscillation).
// 5. Feedforward + PID: add a feedforward term from the planned curvature
//    (steering ≈ wheelbase / radius_of_curvature) so the PID only corrects residuals.
// 6. Model Predictive Control (MPC): optimise a control sequence over a receding
//    horizon (e.g. 1 second), explicitly handling actuator limits and trajectory
//    shape — eliminates the lag inherent in pure feedback control.
// 7. Increase update rate: reduce update_point_thresh further (4→2) so the C++
